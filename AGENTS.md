# Project Operating Rules

## Project Snapshot

- This is a small Next.js App Router project generated from `create-next-app`.
- Runtime stack: Next.js 16.2.10, React 19.2.4, TypeScript 5, Tailwind CSS 4, ESLint 9 flat config.
- Application routes live directly under `app/`; there is no `src/` directory currently.
- Static assets live under `public/`.
- The root layout is `app/layout.tsx`; the home route is `app/page.tsx`; global styles are in `app/globals.css`.
- `CLAUDE.md` imports this file, so keep general agent instructions here rather than duplicating them.

## Security And File Access

- Never read, list, search, summarize, or infer contents from `.git/**`, `.env`, `.env.*`, appsettings files, SSH keys, certificate/key files, or paths containing `secret`, `secrets`, `credential`, or `credentials`.
- The only allowed env-like examples are `.env.example`, `.env.template`, `.env.sample`, and `.env.dist`.
- Use safe git commands only, such as `git status --short`, `git branch --show-current`, and `git log --oneline -n 5`; never inspect files inside `.git`.
- Do not introduce secrets into source, logs, generated docs, screenshots, or test fixtures.

## Development Workflow

- Use npm because this repo has `package-lock.json`.
- Common commands:
  - `npm run dev` starts local development.
  - `npm run build` verifies the production build and TypeScript integration.
  - `npm run lint` runs ESLint via the ESLint CLI; do not use `next lint` because it was removed in Next.js 16.
- Before changing Next.js behavior, read the relevant local docs under `node_modules/next/dist/docs/`, especially App Router guides and API references.
- Prefer focused changes and avoid unrelated refactors in this early-stage template.

## Next.js App Router Rules

- Treat pages and layouts as Server Components by default.
- Add `"use client"` only to the smallest component that needs state, event handlers, effects, browser APIs, or client-only hooks.
- Keep server-only data access out of Client Component import graphs. Use `server-only` if server-only modules are introduced.
- In App Router, folders define route segments; a route becomes public only when a `page.tsx` or `route.ts` file exists.
- Use route groups like `(marketing)` or `(app)` for organization without changing URLs.
- Use private folders like `_components` or `_lib` for implementation details colocated inside `app/` when useful.
- For dynamic routes, remember that App Router route props such as `params` and `searchParams` may be asynchronous in current Next versions. Confirm the exact convention in the bundled docs before coding.
- If enabling Cache Components later, add `cacheComponents: true` in `next.config.ts` and follow the `use cache`, `cacheLife`, `Suspense`, and runtime API guidance from the local docs.

## Styling

- Tailwind CSS 4 is configured through `@tailwindcss/postcss` in `postcss.config.mjs`.
- Keep Tailwind imported from `app/globals.css`; use global CSS only for truly global tokens/base styles.
- Prefer Tailwind utility classes for component styling. Use CSS Modules only when scoped custom CSS is clearer than utilities.
- Keep CSS import order deliberate; do not add import sorting that changes stylesheet order automatically.

## TypeScript And Imports

- Keep TypeScript strictness intact; do not relax `strict`, `noEmit`, or build-time type checks to hide errors.
- Use the `@/*` path alias for project-root imports when it improves clarity.
- Do not edit `next-env.d.ts`; it is generated by Next.js.
- If route typing becomes important, consider enabling `typedRoutes` in `next.config.ts` and include generated `.next/types` as already configured in `tsconfig.json`.

## Verification Expectations

- For code changes, run `npm run lint` at minimum.
- Run `npm run build` for changes touching routing, data fetching, metadata, Next config, TypeScript config, or production behavior.
- For UI changes, manually inspect responsive behavior in the browser when a dev server is available.
- Document any verification command that could not be run and why.
